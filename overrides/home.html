// how reversense works

Four phases into one representation.

Reversense runs as a virtual test bench. Each phase feeds a single universal representation — the model every tool on the platform reads from and writes to.

PHASE 01

Discovery

Profiles the execution environment — permissions, CPU, TEE — provisions the test bench and retrieves the target.

→
PHASE 02

Modeling

Builds the universal representation: code, data, libraries, syscalls, UI, traces — all in one place.

→
PHASE 03

Instrumentation

Auto-generates hooks and probes for every point of interest, then collects and tags execution traces.

→
PHASE 04

Execution & test

Runs the app, enriches the model, then scans with SAST / DAST / IAST or lets you explore by hand.

Universal representation

Continuously enriched automatically as tools and robots discover new data — so nothing you learn at runtime is ever lost.

// the platform

Static and dynamic, in one model.

Reversense pairs static analysis with dynamic instrumentation — what it calls stereoscopic analysis — and keeps every result explorable side by side.

representation

Universal model

Machine code, binaries, scripts, dumps, network traces and syscalls — find everything in one place, kept in sync.

search

Merlin engine

Traverse traces and ASTs across 100+ node types, filter with 400+ built-in tags, and search by adjacency or taint.

decompiler

Radare2, built in

Radare2 ships as the default NativeAnalyzer. Swap in another engine through an adapter whenever you need to.

dynamic

Instrumentation

Theme hooks for Java, Objective-C and native code in a few clicks. KeyPoints load, unload and trigger actions on demand.

deobfuscation

Countermeasure assessment

Automated countermeasure assessment covering anti-hook, anti-root and anti-emulator tactics, plus on-the-fly decompilation of dynamically loaded code.

teamwork

Built to collaborate

One project, X devices, Y analysts — work the same binary at once on separate terminals, no conflicts.
